# 甲号証管理アプリ 仕様書

> このファイルは、ユーザから提供された原仕様書（`koshou_kanri_prompt.md`）の保存版です。
> 実装は本仕様書の章立てに従って進める（§17 推奨順序）。
> 不明点は §0.2 の確認事項を優先して質問する。

## 章立て（要約・主要ポイント）

- **§0 ドキュメント概要 / 同梱資料 / 不明点扱い**
- **§1 プロジェクト概要** — 目的・利用シーン・動作環境（Windows + ローカル Web）
- **§2 技術スタック** — FastAPI / python-docx / docxcompose / React + Vite + Tailwind
  - 結合は `docxcompose` 必須（python-docx 単体は壊れる）
  - 分割は同梱 `split_evidence_docx.py` を使用（**改変禁止**）
- **§3 プロジェクト構造**
- **§4 データフォルダ構造**
  - ルート直下に `甲号証リスト.docx` / `個別マスタ/` / `結合甲号証/` を自動補完
  - 個別マスタのファイル名は **全角3桁** + 任意の枝番 `その<全角>`
  - 結合甲号証は `結合甲号証_<YYYYMMDD-HHMMSS>.docx`
  - バックアップは `_backup/<timestamp>/...`、最新10世代
- **§5 表記ゆれ正規化**（最重要）
  - 全角3桁 + 枝番形式に統一
  - §5.4 のテストケース全通過が必須
- **§6 結合甲号証 → 個別マスタ分解**
  - `【甲第xxx号証】` を区切りマーカーとして検出
  - `vendor/split_evidence_docx.py` を呼び、仮ファイル名 → 全角3桁にリネーム
  - dry-run / overwrite / バックアップ対応
- **§7 個別マスタ → 結合甲号証**（`docxcompose.Composer.append`）
- **§8 甲号証リスト関連**（編集ボタン / 個別マスタから自動生成 / 結合甲号証から自動生成）
- **§9 証拠説明書テーブル** — メタデータは「**本文先頭メタブロック方式**」採用済み（ユーザ確認済み）
- **§10 案件ファイル解析** — 本文・表中の甲号証表記を抽出
- **§11 ルートフォルダ初期化**
- **§12 API 設計** — 11 エンドポイント、dry-run / 共通エラー形式
- **§13 UI 統合** — `v01-UI.md` ベース。無ければ §13.5 フォールバック
- **§14 テスト** — fixtures 自動生成、splitter / combiner / case_parser をテスト
- **§15 起動スクリプト** — `start.bat`、ポート 8765 / 5173
- **§16 重要な実装上の注意（必読）**
  - split_evidence_docx.py は改変禁止
  - docxcompose で結合
  - 全角3桁統一
  - dry-run / バックアップ
  - 不明点は推測で進めない
  - Windows パス・日本語ファイル名・UTF-8
  - エラーメッセージは日本語
- **§17 実装の進め方（推奨順序）**
- **§18 想定 Q&A**
