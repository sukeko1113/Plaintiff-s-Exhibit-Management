# プロジェクトルール

## UI保護(最重要)
- 既存のUIコンポーネント(JSX構造・className・レイアウト・色・余白)は変更禁止
- 機能追加でUI変更が必要な場合は、実装前に必ずユーザーに確認すること
- `v02-UI.md` をUI仕様の正本とする
- ボタンの配置・順序・ラベル文言の変更は禁止

## 開発方針
- 仕様は `SPEC.md` に従う
- 機能追加は既存ファイルを編集せず、新規ファイルで追加することを優先
- 変更前に「何をどのファイルでどう変えるか」を提案してから実装する

## 技術スタック
- Backend: Python / FastAPI / python-docx / docxcompose
- Frontend: React
- 出力形式: Markdown と Word(.docx)

## 禁止事項
- 実際の訴訟データフォルダ(Google Drive側)へのアクセス・参照は禁止
- 開発はこのフォルダ内で完結させること

## SPEC遵守
- 仕様変更が必要な場合、まずSPEC.mdの該当箇所を提示し、ユーザー承認を得てから実装する
- 勝手にAPIエンドポイントを増やさない
- 既存の10個のエンドポイント(SPEC.md参照)を変更しない

## PRの作り方
- 1つのPRには1つの機能のみ含める
- PRの説明欄に「変更したファイル一覧」と「UI変更の有無」を明記する
- 既存のテスト(11個のテストケース)を壊さないこと

## 不明点の扱い
- 仕様が曖昧な場合、推測で実装せず必ずユーザーに質問する
- 特にメタデータ欄(タイトル・日付・著者・立証趣旨)の取得元については慎重に確認する
